# CONSTITUTION - Engineering Governance for Multi-Agent Development

This file is the **immutable source of truth** for repository governance.

It defines the non-negotiable operating law for agentic work on this repository.

If any lower-level artifact conflicts with this file, this file wins.

Do not edit this file during normal feature execution.

---

## Purpose

This constitution exists to make repository work:

- bounded
- reviewable
- revertable
- auditable
- safe under agent execution

Its purpose is not bureaucracy.

Its purpose is to prevent speed from turning into silent structural debt.

---

## Constitutional Precedence

When sources conflict, use this order:

```text
CONSTITUTION > AGENTS.md > NLSPEC > STATE > DECISIONS > TODO > verbal instruction
```

### Meaning

- `AGENTS.md` cannot weaken constitutional rules
- `STATE.<slug>.md` cannot override repo law
- `DECISIONS.<slug>.md` cannot justify scope expansion
- `TODO.<slug>.md` cannot redefine what is allowed
- verbal instructions never outrank written governance

If ambiguity remains, escalate. Do not improvise.

## Constitutional Mirrors

Each initialized repository must contain both:

- `.opencode/_constitution.md` -> immutable source of truth
- `docs/governance/constitution.md` -> human-readable mirror

If the mirror conflicts with this file, this file wins.

If the mirror is missing, coding must not begin.

## Prime Law

### 1. No execution without an explicit contract

No `STATE.<slug>.md` -> no coding.

### 2. No execution without a valid worktree

No dedicated worktree -> no coding.

### 3. No execution on protected branches

Never implement directly on:

- `main`
- `master`
- `develop`
- `trunk`

### 4. No self-expanding scope

If implementation requires more scope than the current contract allows:

- stop
- return to planner
- update `STATE.<slug>.md`
- satisfy any newly required gates
- re-run preflight before resuming

### 5. One active task at a time

`TODO.<slug>.md` must contain exactly one active current task before coding begins.

### 6. One task = one commit

Every completed task must produce one atomic commit immediately.

### 7. Immutable control-plane assets remain immutable

These files must never be edited during ordinary feature execution:

- `.opencode/_constitution.md`
- `.opencode/_STATE.md`
- `.opencode/_TODO.md`
- `.opencode/_DECISIONS.md`
- `.opencode/skills/*`

### 8. Required gates must be satisfied before execution

A gate is not satisfied because it exists in a flow.  
It is satisfied only when the required outcome or approval actually exists.

### 9. Stability beats improvisation

Blocked execution is preferable to unsafe execution.

## Repository Initialization Law

`AGENTS.md` may begin life as a template.

But once the repository is initialized for real work:

- all required project configuration sections must be concretely filled
- unresolved template placeholders are forbidden
- preflight must block coding if `AGENTS.md` remains in template state

Examples of forbidden unresolved placeholders:

- `<!-- FILL -->`
- `<!-- path -->`
- `<!-- I-NN -->`
- equivalent repo-template markers

No initialized AGENTS -> no coding.

## Authority Order

```text
$governance > $architect-security > $architect > $security > $qa > $review
$planner (scope, no veto) | $preflight (gate, no veto) | $triage (routing, no veto)
$coder (execution, no veto)
```

## Authority Meaning

### `$governance`

Protects:

- constitutional law
- invariants
- public contract policy
- migration/rollback bypass
- conflicts between higher-order rules

### `$architect-security`

Protects combined:

- structural risk
- security/trust risk

### `$architect`

Protects:

- boundaries
- system shape
- undefined blast radius
- structural escalation

### `$security`

Protects:

- auth
- secrets
- dependencies
- network exposure
- untrusted input
- trust surfaces

### `$qa`

Protects:

- acceptance criteria satisfaction
- regression confidence
- proof quality

### `$review`

Protects:

- scope discipline
- merge discipline
- atomicity
- traceability
- final repo integrity before release gate

### `$planner`

Owns the feature execution contract.

### `$preflight`

Owns final execution readiness checking.

### `$triage`

Owns request classification and flow selection.

### `$coder`

Is the only skill allowed to write production code.

## Feature Contract Law

Every real implementation change must be governed by:

- `AGENTS.md`
- `STATE.<slug>.md`
- `TODO.<slug>.md`
- `DECISIONS.<slug>.md`

### Artifact Meaning

- `AGENTS.md` = routing + repo-level configuration
- `STATE.<slug>.md` = feature contract
- `TODO.<slug>.md` = execution rail
- `DECISIONS.<slug>.md` = feature-local memory for non-trivial decisions

If any required artifact is missing, stale, contradictory, or incomplete:  
implementation must not proceed.

## Change Level Law

All repository work must be classified into one of:

- `L1` - local low-risk
- `L2` - bounded standard change
- `L3` - structural or sensitive

If uncertain, choose the higher level.

The level controls:

- rigor
- evidence expectations
- required gates
- acceptable flow

See `docs/governance/levels.md`.

## Canonical Flow Law

### Standard feature

`triage -> planner -> preflight -> coder -> qa -> review -> doc -> release`

### Bug fix

`triage -> planner -> preflight -> coder -> qa -> review`

### Structural change

`governance? -> triage -> planner -> architect -> adr -> preflight -> coder -> qa -> review -> doc -> release`

### Security-sensitive change

`governance? -> triage -> planner -> architect-security -> adr -> preflight -> coder -> security -> qa -> review -> doc -> release`

### Canonical Rule

If a constitutional or invariant surface is touched, governance comes first.

## Preflight Placement Law

Preflight must occur:

- after planning
- after all required upstream gate satisfaction
- immediately before coder

Incorrect ordering such as `preflight -> planner` is forbidden.

## Hard Execution Gates

Coding must not begin if any of the following are true:

- `AGENTS.md` is still in template state
- `.opencode/_constitution.md` is missing
- `docs/governance/constitution.md` is missing
- `STATE.<slug>.md` is missing
- `STATE.<slug>.md` is stale or incomplete
- `TODO.<slug>.md` is missing
- `TODO.<slug>.md` does not contain exactly one current task
- `DECISIONS.<slug>.md` is missing
- required gates are unsatisfied
- collision risk is unresolved
- inter-skill conflict is unresolved
- execution is on the wrong branch or outside a dedicated worktree
- blast radius is unknown

## Scope Discipline Law

`STATE.<slug>.md` is the feature contract.

Anything not explicitly in scope is out of scope.

If coder or any downstream step discovers:

- additional required files
- larger blast radius
- changed public contract impact
- architecture tension
- new trust/security implications

then execution must stop and return to planner.

No downstream artifact may silently enlarge the contract.

## Refactor Law

Refactoring is forbidden unless explicitly approved by the contract and required gates.

Forbidden during normal feature work:

- opportunistic renames
- unrelated cleanup
- directory reshuffling
- cross-cutting modernization
- abstraction introduction without contract need
- "while I was here" structural edits

If refactor pressure appears:

- stop
- return to planner
- escalate when structure is involved

## Decision Logging Law

Non-trivial local implementation decisions must be recorded in `DECISIONS.<slug>.md`.

But `DECISIONS.<slug>.md` must not be used to:

- expand scope
- bypass gates
- settle architecture/invariant/contract questions locally

If a decision affects:

- architecture
- invariants
- public contracts
- trust boundaries
- migration/rollback assumptions
- schema/config/file format compatibility

it must be escalated to the correct gate and, when required, to ADR.

## ADR Law

ADR is required when applicable for:

- invariant changes
- architectural boundary changes
- public contract changes
- config/schema/file format compatibility decisions
- runtime or pipeline semantic changes
- trust boundary changes
- migration-sensitive decisions
- rollback-sensitive decisions
- long-lived architecture policy

If ADR is required, implementation must not proceed until the ADR gate is satisfied.

## Documentation Law

Documentation is required when changes affect:

- user-visible behavior
- operator-visible behavior
- config usage
- public API / CLI expectations
- architecture understanding
- workflows or runbooks
- migration or compatibility understanding

Documentation must describe approved reality, not hopeful intent.

## Validation Law

Proof requirements are real requirements.

- unproven acceptance criteria are not complete
- compile success is not acceptance
- weak adjacent validation is not strong proof
- bug fixes require regression confidence appropriate to risk
- higher levels require stronger evidence

QA must not pass unproven work.

## Commit Discipline Law

### Atomicity

One task = one commit.

### Timing

Commit immediately after task completion.

### Format

Commit format must be:

```text
type(scope): description
Task: T-NNN
```

### Traceability

Every completed task in `TODO.<slug>.md` must end with:

`| commit: <short-SHA>`

This exists for:

- auditability
- reviewability
- revertability
- task/commit traceability

Broken traceability is a real defect.

## Drift Law

Execution must stop immediately when drift appears.

Drift includes:

- scope expansion
- need for out-of-scope files
- forbidden area becoming necessary
- architecture tension
- changed public contract impact
- changed required gates
- larger blast radius
- invalidated plan assumptions

When drift appears:

- stop coding
- return to planner
- update STATE
- satisfy any newly required gates
- re-run preflight before continuing

No "just finish this part first."

## Parallel Work Law

Parallel work is allowed only when collision risk has been checked and accepted.

High-risk shared surfaces include:

- public interfaces
- schemas
- config
- pipelines
- shared core modules
- trust boundaries

Unresolved collision risk blocks execution.

## Final Gate Law

A change is not merge-ready or release-ready merely because code exists.

Final readiness requires:

- required gates satisfied
- contract complete
- QA complete where required
- review approval
- docs complete where required
- ADR complete where required
- TODO/commit traceability intact
- no open constitutional violation

`$release` may declare:

- `MERGE_READY`
- `RELEASE_READY`

only after all required obligations are complete.

## Non-Negotiable Summary

- No initialized AGENTS -> no coding
- No STATE -> no coding
- No worktree -> no coding
- No single current task -> no coding
- No satisfied required gates -> no coding
- No constitution mirror -> no coding
- Unresolved conflict -> no coding
- Unknown blast radius -> no coding
- Drift -> stop
- One task = one commit

Stability is a feature.
